
# 1 文档建立
我们开始这个项目之前，请新建几个文档
每个文档的内容你可以和我一起沟通制定

## 1.1 开发环境
- 描述开发的主机配置；
- 描述开发所使用的工具软件；
- 描述安装目标的配置；
- 描述是否具备网络IP，域名等远程连接地址与端口；
- 描述开发过程中，需要使用的相关账号，token，API地址，key等信息；
- 描述交付后，因使用所需要配套具备的账号，token，API地址，key等信息；
- 用户提供的参考文档，如API接口，项目地址等；

## 1.2 需求文档
记录用户所提出来的需求，分为
1. 用户原始需求；
2. AI 理解并修改或者添加内容；

*需求文档需要根据用户在调试过程中提出来的需求变更，新增等进行同步优化*

## 1.3 开发设计文档
记录面向需求所使用到的框架，接口，功能设计相关描述，需要与需求文档的内容同步进行更新，避免遗漏；
包括
1. 总体设计架构图；
2. 所使用的接口；
3. 交互逻辑；
4. 等；

*开发设计文档需要根据用户在调试过程中提出来的需求变更，并确定后进行对应内容同步优化*

## 1.4 部署文档
该文档在第一次交付时候，根据开发内容进行填充内容，用于记录用户使用时候所需要的相关信息，方便用户自己搭建，或者转交他人搭建，

信息包括如
1. 服务器，如windows，linux
2. 运行环境，如docker，
3. 部署方法；
4. 备份恢复方法；


# 2 开发规范与流程
开发过程中，应该遵守如下规则

## 2.1 用户沟通
1. 当需要与用户进行沟通时候，请用简单明了的语言描述；
2. 当需要想用户解释架构，技术细节时候，请使用简单的语言进行阐述；
3. 当用户不明白技术时候，请提供不同的方案，并说明优缺点，让用户进行选中，并告知是否有风险；

## 2.2 版本规范
初始版本号为V0.0.0, 开始递增
1. debug修复问题，优化的，请更新最后一位数；
2. 用户提出的新需求，请更新第二位；
3. 用户主动提出进行大版本更新的，或者设计重大架构变更的，请更新第一位；

## 2.3 需求更新
1. 用户在过程添加的需求，请理解并及时更新到需求文档.md中，避免事后遗漏；
2. 更新相关需求文档后，需要对应更新对应有变化的开发设计文档，部署文档，开发环境文档；
3. 如果对需求有不明白的，可以找用户进行沟通确定；
4. 如果有更好的方案，请主动提出，让用户选中；

## 2.4 代码实现
1. 如果在代码实现过程中，出现开发设计和需求与实际代码不同，请及时更新相关文档；
2. 代码实现过程中，如果发现技术不可行，或者有更好的方案，请及时与用户沟通确定；


# 3 封板
1. 当用户说封板，则进行git以及上传github，并创建release版本，需要具备交付的文件；
2. 封板时候，自动整理与上个封板版本的差异，并检查需求文档与开发设计文档中是否具备对应内容，如果没有，请更新需求文档与开发设计文档，开发环境等文档中；
3. 当实际功能与文档不符时，请询问用户以哪个为准；
